- 2026-07-16 — **the axis of the front-page sample is labelled with dates that hold no data, and it has shipped since v1.** Chart.js picks `unit="day"` for every monthly series — measured 3, 6 and 12 monthly points, all `day`. The bundled `monthly-sales.csv` (12 points, Jan 1…Dec 1) renders ticks at `Jan 1, Feb 12, Mar 26, May 7, Jun 18, Jul 30, Sep 10, Oct 22`; only the first is a data point. Three months renders `Jan 1, Jan 9, … Feb 26` — 8-day ticks — and the Mar 1 point falls past the last label. v1.3 set `time.unit` **only** when `yearAxis` is true (`chart.ts:253`), so it killed the "Feb 2019 / Aug 2019" labels for years and left the identical bug for months. Root cause is one branch, not a Chart.js quirk we can't reach.
- 2026-07-16 — **why nobody caught the axis bug for four releases: `docs/hero.png` is weekly data.** Weekly points sit on a fixed 7-day grid, and day-unit ticks are also a fixed grid, so its 14-day ticks land exactly on every second data point — the hero is correct. Months are 28–31 days, so day-unit ticks drift off the 1st. The one chart anybody ever actually looked at was the one kind of data that cannot show the defect. Worth remembering when picking a sample or a demo: a fixture that can't fail isn't a fixture.
- 2026-07-16 — **process lesson, promoted to AGENTS.md Core Rule 2: numbers about the output are not the output.** 16 reviewers (including a data-viz expert who computed BT.601 luma of the palette, and a grad student who measured exported SVG coordinates), 120 green tests, and a whole session of my own measurements — contrast ratios, tap targets, CSP violation events, PNG IHDR dimensions — all missed an axis labelled with dates that hold no data. Every measurement was correct and every one was of the wrong thing: `timeAxis` **was** true, detection **was** right. No test asserts a tick label; the SVG smoke test only checks that `<path>` exists. It surfaced within a minute of rendering the PNG and opening it, and only because the user pushed me to look at the screen instead of reporting numbers. For a tool whose product is an image, evidence that never contains the image is a blind spot with a process built around it.

- 2026-07-16 — dates: gate `Date.parse`'s *input* rather than removing `Date.parse`. A closed list of shapes (ISO dates/instants, `YYYY-M-D HH:MM`, `YYYY/M/D`, `Mmm YYYY`, `D Mmm YYYY`, `Mmm D, YYYY`) may reach it; everything else is a category. `Date.parse` is spec-defined for ISO and engine-defined only for the rest, so this closes the guessing without losing the standardised half — the first draft's "no date reaches `Date.parse`" would have taken every ISO timestamp with it and drawn irregular samples as equal-width bars. The regression fixture (`test/dates.test.ts`) was written and run green **before** the parser changed, then re-run after: that is the guardrail the pre-mortem asked for, and it immediately earned its keep by catching that `Jan 5, 2025` unquoted is two CSV fields, so the fixture had been passing on `Date.parse("Jan 5")` = 5 Jan 2001 — the very bug under test.
- 2026-07-16 — encoding: `repl >= 2 && repl > valid`, an absolute floor plus a comparison, not a ratio. Measured: Big5-header-plus-1000-rows, Latin-1, UTF-16LE (all must refuse) and a good UTF-8 file with one stray byte (must render) **all score ratio 1.000**, so no threshold of that shape can separate them — only the count can. The floor of 2 buys the CP1252-smart-quote case (one stray byte out of Word, the commonest near-UTF-8 artifact) at the price of Latin-1 files with exactly one accent, which are provably undecidable — identical in (repl, valid) to that stray byte. Rendering a good file matters more than catching an undecidable one; refusing a file that *is* UTF-8 would have the tool tell the user a falsehood about their own data, with no override.
- 2026-07-16 — tripwire: run the full suite on the deploy path, not the "hermetic tests gate deploy, canvas tests gate merge" split this SPEC first specified. The split was borrowed from a pre-mortem worried about apt-get on the release path, and it does not survive contact with this repo: solo maintainer, direct pushes to `main`, no branch protection — so "gates the merge" gates nothing, and a red SVG smoke test would still deploy, which is the whole problem. Cost: a duplicate apt+test on `main`. - 2026-07-10 — v1.3 bare years now require a temporal header (`year`, `yr`, `年`). The expert review showed the old rule turned a `count` column of 2000/2010/2020 into a time axis, and charted 1898–1902 differently from 2018–2022 purely because of the 1900–2099 window. A column that *is* bare years also gets `time.unit: "year"`, which finally kills the "Feb 2019 / Aug 2019" labels — Chart.js otherwise only picks year labels once the span covers numTicks-1 years.
- 2026-07-10 — v1.3 SVG ids: canvas2svg names clipPaths with a fresh random string each render, so the same CSV exported twice differed byte-for-byte. Renamed in document order (cs0, cs1…). - 2026-07-10 — v1.3 blank rows: dropped `skipEmptyLines: "greedy"` for our own filter. Greedy ate a header row of blank names (`,,`), quietly promoting the first data row to the header. A blank *line* carries no fields and is noise; a row of blank *cells* has structure. Blank lines around and inside the data are still ignored.

- 2026-07-10 — promoted the security review's throwaway test to a permanent suite (test/security.test.ts) — the exported SVG is a file users reopen in a browser, where `<script>` executes, so canvas2svg's DOM-based escaping is load-bearing and deserves a tripwire. Also covers the `String.replace` `$&`/`$1` hazard in the EXPORT_BG injection (user data never reaches the replacement argument) and a NUL cell (fail-closed: malformed XML, never script). jsdom's XMLSerializer duplicates `xmlns:xlink` on the root; verified against a real browser that this is a jsdom artifact, so the test normalizes it rather than chasing a phantom.
- 2026-07-10 — v1.2 number parsing: detect the US/EU convention **per column**, not per cell — a single cell can never settle "3.850". Decisive evidence: two separators (last one is the decimal), 2+ dots (grouping → EU), 2+ commas (grouping → US), or a separator in a non-grouping position. When nothing is decisive we keep the American reading — so no existing chart silently moves — and emit a note naming both readings. Chose a note over a guess because a wrong chart that looks right is this tool's worst failure mode. `2,500` is NOT flagged (a comma with exactly three decimals is vanishingly rare, and it's Excel's default US export); `3.850` IS, because a dot-thousands separator is the actual reported bug. Only grouping-position separators are stripped: stripping every comma turned "1.234,56" into 1.23456 — a plausible-looking wrong number rather than an honest gap.
- 2026-07-10 — v1.2 dates: parse bare `YYYY`, `YYYY-MM`, `YYYY-MM-DD` as LOCAL midnight rather than letting `Date.parse` apply the ISO UTC rule, because the time axis draws in local time — a New York user's `2025-01-01` was rendering as the tick "Dec 31". Cells carrying a time or an explicit offset keep their exact instant. Side benefit: ticks now land on the data's own months (the sample's axis starts at "Jan 2025", not "Feb 2025"). Impossible dates (`2025-02-31`) are rejected instead of rolling into March.
- 2026-07-10 — audit of the adapter swap, from several user angles. Confirmed safe: SVG export escapes hostile column names (canvas2svg serializes text through the DOM — `<script>` lands as `&lt;script&gt;`, no event-handler attributes, parses as valid XML), the live DOM only ever gets `textContent`, and PNG export is byte-identical across runs (SHA-256). Confirmed broken, all PRE-EXISTING and all deferred to SPEC "Later": (a) EU `3.850` parses as 3.85 — 1000x off, silently, chart still looks plausible; the csv.ts TODO understated this as "reads as a category", which is only true for the `1.234,56` form; (b) a 7th+ numeric column is silently dropped (`yColumns.slice(0, 6)`) with no note; (c) `2025-01-01` is parsed as UTC midnight but rendered on a local-time axis, so a New York user sees the tick "Dec 31" — the same CSV renders differently per timezone, which contradicts the deterministic wedge; (d) SVG export is not byte-deterministic because canvas2svg mints random clipPath ids (geometry is identical once ids are stripped); (e) "works offline after first load" has no service worker behind it (nothing in src/, index.html or vite.config.ts registers one) — it rests entirely on whatever the browser HTTP cache happens to retain; the actual Pages cache headers were not measured.
- 2026-07-10 — year-only CSVs (`year,value`) label the x axis "Feb 2019"/"Aug 2019" — pre-existing, NOT caused by the adapter swap: Chart.js only labels by year when `diff(max,min,'year') >= numTicks-1` (4 >= 7 is false), and both adapters return 4 there. Confirmed the span is what matters: 26 years of data labels cleanly as 2001…2025. Deferred to SPEC "Later" rather than fixed inside an optimization pass.
- 2026-07-10 — `diff` for day/week must count calendar days, not `(max-min)/86_400_000` — a local day is 23 or 25 hours around a DST switch, so the naive version was off by one for any range crossing one. Caught only on re-audit: the first differential run passed because this machine is UTC+8 (no DST), so its "DST stamps" crossed nothing. Re-run pinned to America/New_York, Europe/Berlin, Australia/Sydney and Pacific/Chatham (UTC+12:45). Regression tests pin TZ themselves, since CI runs UTC; verified they fail against the old implementation before trusting them.
- 2026-07-10 — replaced chartjs-adapter-date-fns + date-fns with a hand-rolled adapter (src/date-adapter.ts) — 48.6 kB of bundle spent on labelling ticks. Differential-tested against date-fns (every unit, leap/DST/year-end stamps, five timezones) while it was still installed. Two divergences, both deliberate: (1) date-fns nudges late-February dates to the 30th inside differenceInMonths; we count plain calendar months, which can only shift which unit labels the axis when a bound lands on Feb 28–29. (2) The old adapter's `aaaa` token rendered "1:45 p.m."; we render "1:45 PM", which reads better on a chart axis — sub-hour time series are the only ones affected. Month names are hardcoded English rather than Intl: a locale-aware axis would break the "same CSV always renders the same chart" promise.
